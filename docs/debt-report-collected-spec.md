## การแสดงผลแถวแบ่งชำระ

- เมื่อมีการชำระมากกว่า 1 ครั้งในงวดใด ๆ จะมีแถวเพิ่มต่อจากแถวสัญญาหลัก
- คอลัมน์ซ้าย 9 คอลัมน์: คอลัมน์ "ชื่อ-นามสกุล" จะถูกแทนด้วยข้อความ `- แบ่งชำระ -` (italic, สีจาง), ส่วน วันที่อนุมัติ / เลขที่สัญญา / เบอร์โทร / ยอดผ่อนรวม / งวดผ่อน / ผ่อนงวดละ ยังแสดงซ้ำเดิม, สถานะ + เกินกำหนด เว้นว่าง
- คอลัมน์ขวา: เฉพาะงวดที่มีการแบ่งชำระจริง จะแสดง row ของครั้งที่ 2/3/... ใน column block ของงวดเดียวกัน (column `งวดที่` เว้นว่าง, ส่วนเงินต้น/ดอก/ฯลฯ ใส่เฉพาะส่วนของครั้งนั้น)
- ในตัวอย่าง reference สังเกต: contract CTR-260-0001 มี 1 sub-row, contract CTR-260-0003 มี 3 sub-rows

## เงื่อนไขการคำนวณ (ตามที่ผู้ใช้แจ้ง)

1. **ชำระเกิน (overpaid)**: ถ้าลูกค้าจ่ายเกินยอดงวด → ส่วนเกินถูกบันทึกไว้ที่งวดนั้น (column "ชำระเกิน") และนำไปลด **เป้า** (target) ของงวดถัดไป
   - ลำดับการตัด: ค่าดำเนินการ → ดอกเบี้ย → เงินต้น (จนกว่าส่วนเกินจะหมด)
   - ตัวอย่าง: CTR-260-0020 งวด 1 จ่าย 12,400 (target 9,400) → overpaid 3,000 → งวด 2 ของแถบ "เป้าเก็บหนี้" จะลดลง 3,000

2. **ปิดค่างวด (close-installment)**: ลูกค้าจ่ายยอดที่เหลือทั้งหมดมาในครั้งเดียว
   - column "ปิดค่างวด" ของงวดนั้น = ผลรวมเงินต้นของงวด N+1..end
   - งวด N+1..end จะไม่มีเป้าเก็บหนี้ในแถบ "เป้าเก็บหนี้"
   - ตัวอย่าง: CTR-260-0002 งวด 6 มี ปิดค่างวด = 141,120 (=18 งวดที่เหลือ × 7,840)

3. **หนี้เสีย (bad-debt)**: ยอดที่ขายเครื่องคืนได้ → บันทึกที่ "งวดล่าสุด" ของยอดเก็บหนี้
   - column "หนี้เสีย" ใส่ยอดที่ได้จากการขาย
   - ตัวอย่าง: CTR-260-0024 inst 32 บันทึก 15,000 (เงินต้น/ดอก/ดำเนินการ ทั้งหมด = 0)
   - "ยอดที่ชำระรวม" = หนี้เสีย เพราะไม่มียอดอื่น

## คอลัมน์เพิ่มจาก reference (ที่อาจเป็น 0 ทั้งคอลัมน์)

- **ค่าปรับ** (penalty late fee): partner Boonphone อาจไม่เก็บแยก → จะเป็น 0 เกือบทุกแถว
- **ค่าปลดล็อก**: ค่าธรรมเนียมเมื่อลูกค้าผิดนัดถึงขั้นต้องล็อกเครื่อง — ปกติเป็น 0
- **ส่วนลด**: ส่วนลดที่บริษัทให้ — ปกติเป็น 0

## หมายเหตุการ map กับ DB ของเรา

ตาราง `payment_transactions.raw_json` มี field:
- `payment_amount` = ยอดที่ชำระจริง
- `payment_date` = วันที่ชำระ
- `principal`, `interest`, `processing_fee` (อยู่ใน `breakdown` หรือคำนวณจาก `installment.principal/interest/processing_fee`)
- `late_fee`, `unlock_fee`, `discount`, `overpaid`, `close_amount`, `bad_debt_amount` ต้อง map จาก field พิเศษ — ถ้า partner API ไม่ส่งให้ ตั้งเป็น 0

ต้องสร้าง logic ฝั่ง server ที่:
1. ดึง payments ทุกครั้งของสัญญา
2. group ตาม `installment_no`
3. หา `principal/interest/fee` จากการแบ่ง payment_amount ตาม schedule
4. คำนวณ overpaid/close/bad-debt ตามเงื่อนไขข้างบน
5. Output array `installments[].payments[]` (1 หรือมากกว่า) ให้ frontend render เป็นแถวหลัก + แถว `- แบ่งชำระ -`
